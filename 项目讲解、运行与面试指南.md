# CSTCloud-RAG 项目讲解、运行与面试指南

> 目标岗位：AI 应用开发、RAG 开发、大模型应用开发、AI 全栈开发实习。本文完全按当前代码实现撰写，不包含虚构的线上用户量、准确率或性能数据。

## 一、先把项目讲成一句人话

这是一个企业知识库问答系统：用户上传 PDF、Word、Markdown、Excel 等文档，系统将文档解析、语义切片并向量化；提问时同时使用向量检索和 BM25 关键词检索召回证据，再用 Rerank 重排，最后把可靠证据交给中国科技云大模型生成带引用的流式回答。

项目的重点不是“调用一次大模型”，而是把文档处理、索引、召回、融合、重排、生成、引用、会话和可观测性组成一条完整工程链路。

## 二、面试开场话术

### 30 秒版本

“我做了一个基于中国科技云 API 的企业级 RAG 知识库系统。后端使用 FastAPI 和 SQLite，向量库使用 Chroma；前端使用 Vue3 和 TypeScript。文档上传后会按标题、段落和句子语义切分，再调用 Embedding 入库。查询侧不是只做向量检索，而是把向量检索与 BM25 通过 RRF 融合，再调用 Rerank，最后让大模型基于编号证据生成带页码和 chunk_id 的答案。系统还支持多模型切换、Query Rewrite、多轮会话、SSE 流式输出和 Docker 部署。”

### 2 分钟版本

“我把这个项目拆成了两条主链路。第一条是索引链路：支持六类文档解析，保留文档名、页码、章节和 chunk_id；切分时优先按标题和段落，过长内容再按句子处理，并设置 overlap。切片原文存 SQLite，向量写入 Chroma。知识库会绑定创建时的 Embedding 模型，collection 名中加入模型名和哈希，避免模型切换后不同维度的向量混写。

第二条是问答链路：多轮追问先通过 Query Rewrite 改写成独立问题；然后同时执行 Chroma 向量召回与 BM25 关键词召回。因为两类分数尺度不同，我没有直接相加，而是采用加权 RRF 按排名融合；融合结果再交给科技云 Rerank。最终证据被编号后写入严格的 RAG Prompt，模型必须根据资料回答。后端先通过 SSE 返回引用和检索耗时，再逐 token 返回回答，并保存会话。

工程上我统一封装了科技云的 models、chat、embeddings、rerank 接口，处理了 DeepSeek-R1 只接受 user role、不同模型思考参数不一致、请求超时和有限重试等问题。系统没有 Token 也能启动，但真正调用模型时会明确提示配置环境变量。项目目前通过 5 项后端测试、前端 TypeScript 生产构建和无 Token 启动检查。”

## 三、系统架构

```text
Vue3 控制台
  │  REST + POST SSE
  ▼
FastAPI API 层
  ├─ 模型配置 / 知识库 / 文档 / 会话 / 评测
  ├─ 文档解析 → 语义切片 → CSTCloud Embedding → Chroma
  ├─ Query Rewrite
  ├─ Chroma 向量检索 ─┐
  ├─ BM25 关键词检索 ─┼→ 加权 RRF → CSTCloud Rerank
  └─ RAG Prompt → CSTCloud Chat → SSE + citations

SQLite：知识库、文档、原文切片、模型配置、会话与消息
Chroma：按“知识库 + Embedding 模型”隔离的向量 collection
```

### 技术栈为什么这样选

- FastAPI：原生支持异步接口、Pydantic 参数校验和 OpenAPI，适合大模型 API 代理及 SSE。
- Vue3 + TypeScript：组件化管理模型控制台、文档管理、会话和流式消息状态。
- SQLite：展示版零配置，足以保存元数据和会话；生产环境可迁移 PostgreSQL。
- Chroma：本地持久化、metadata 查询和 collection 管理简单，适合个人项目演示。
- rank-bm25：补足向量检索对精确编号、专有名词、日期和文件术语召回不稳的问题。
- httpx：直接实现科技云协议，避免业务逻辑被框架黑盒包裹，面试时也更容易讲清每一层。

## 四、索引链路：上传一份文档后发生了什么

### 1. 文件安全检查

上传入口检查扩展名与文件大小，保存后再根据实际落盘大小二次检查；文件名通过 `Path.name` 去除目录，删除文件时再次验证路径位于上传根目录内。

支持格式：`txt`、`md`、`pdf`、`docx`、`csv`、`xlsx`。

关键代码：`backend/app/services/knowledge_base_service.py`。

### 2. 文档解析

- PDF：使用 pypdf 按页提取文本，页码从 1 开始保存。
- Word：识别 Heading 样式，并额外抽取表格。
- Markdown：识别 `#` 到 `######` 标题。
- CSV/Excel：使用 pandas 转成“字段名: 值”的结构化文本。
- 扫描 PDF：当前版本会明确提示“未解析出文本”，并预留 DeepSeek-OCR 服务边界。

关键代码：`backend/app/services/document_parser.py`。

### 3. 语义切分

默认 `chunk_size=800`、`chunk_overlap=120`。顺序是：

1. 优先保留解析阶段识别的标题和章节信息。
2. 按空行分段。
3. 过长段落按中文句号、问号、感叹号和分号拆分。
4. 单句仍过长时才按字符窗口切分。
5. 只有剩余空间足够时才携带 overlap，保证最终 chunk 不超过配置长度。

这样比纯固定窗口更容易保住一个知识点的上下文。每个 chunk 保存文档、知识库、页码、章节、索引和创建时间。

关键代码：`backend/app/services/text_splitter.py`。

### 4. 向量化与持久化

Embedding 按批次调用科技云接口。原文和 metadata 写入 SQLite，向量和检索 metadata 写入 Chroma。

collection 名称大致为：

```text
kb-{知识库ID前8位}-{规范化Embedding模型名}-{模型名SHA1前10位}
```

这解决了一个常见问题：`bge-large-zh`、`gte-qwen2` 和 `qwen3-embedding` 的向量维度可能不同，不能写进同一 collection。前端检测到当前配置与知识库绑定模型不一致时，会提示全库重建。

关键代码：`backend/app/services/vector_store.py`。

## 五、问答链路：用户提问后发生了什么

### 1. Query Rewrite

多轮对话中的“它支持吗？”不能直接检索，因为“它”缺少实体。系统取最近若干轮上下文，让聊天模型只输出一个信息完整的独立检索问题。

如果改写接口失败，系统退回原始问题，不让整个问答链路失败。

### 2. 向量检索

使用知识库绑定的 Embedding 模型生成问题向量，在 Chroma 中执行余弦距离 top-k 检索。返回值转换为 `1 - distance`，方便界面展示。

### 3. BM25 检索

BM25 适合精确词、型号、日期和政策编号。代码将英文数字词、中文单字和中文二元词组作为 token，建立当前知识库的 BM25 语料并返回 top-k。

### 4. RRF 融合

向量相似度和 BM25 分值没有统一量纲，直接按 `0.6 * vector_score + 0.4 * bm25_score` 相加不稳定。系统根据排名使用 Reciprocal Rank Fusion：

```text
score(d) = 0.65 / (60 + vector_rank(d))
         + 0.35 / (60 + bm25_rank(d))
```

最后再除以最高分进行归一化。权重 0.65/0.35 表示更信任语义召回，同时保留关键词召回的纠错能力。

### 5. Rerank

Embedding 适合大规模初筛，Rerank 会联合读取“问题 + 候选文本”，对细粒度相关性判断更准确，但成本更高。因此系统先召回 `top_k=12`，再默认重排并保留 `top_n=5`。

Rerank 失败时会降级使用 RRF 排名，不影响回答链路继续执行。

### 6. Prompt 与引用

证据以如下格式交给模型：

```text
【资料 1】
来源：产品手册.pdf，第 3 页，chunk_id=xxx
内容：...
```

系统 Prompt 约束模型只能依据资料回答，资料不足时必须说“知识库中没有找到足够依据，无法确认”。后端同时返回结构化 citations，包含文件名、页码、chunk_id、原文、融合分数和 rerank 分数。

### 7. SSE 流式返回

后端不是只传字符串，而是发送四类事件：

- `meta`：会话 ID、模型、RAG 是否启用。
- `retrieval`：改写问题、引用、命中数量和检索耗时。
- `reasoning` / `token`：思考内容与最终回答片段。
- `done`：消息 ID、生成耗时和总耗时。

前端使用 `fetch` 读取 POST 响应流并自行解析 SSE，而不是使用只能发 GET 的原生 `EventSource`。

流式请求只在首个 token 产生前重试；一旦已经输出内容，连接中断就直接报错，避免重试导致回答片段重复。

## 六、科技云客户端值得讲的细节

关键代码：`backend/app/clients/cstcloud_client.py`。

- API Key 只从 `CSTCLOUD_API_KEY` 环境变量读取，不进入前端、数据库或日志。
- 统一设置连接超时、读取超时、错误结构和指数退避重试。
- `deepseek-r1` 只接受 user role，因此将 system/history/user 合并为单个 user 消息。
- `qwen3:235b` 使用 `chat_template_kwargs.enable_thinking`。
- `deepseek-v4-flash` 使用 `chat_template_kwargs.thinking`。
- `/models` 获取失败或没有 Key 时，前端展示内置候选，但模型调用会明确返回配置提示。
- Embedding 会检查返回向量数是否与输入文本数一致，避免错位写库。

## 七、数据模型怎么解释

SQLite 中主要有六类对象：

- ModelConfig：聊天、Embedding、Rerank 模型和检索参数。
- KnowledgeBase：知识库名称、描述、绑定的 Embedding 模型。
- Document：原始文件路径、类型、状态、错误信息和 chunk 数量。
- Chunk：原文、页码、章节和所属文档。
- Conversation：会话标题及绑定知识库。
- Message：角色、回答、引用 JSON、模型和耗时。

为什么 Chunk 原文还要存 SQLite，而不是只放 Chroma？因为 BM25、切片检查器、审计、引用展示和全库重建都需要稳定读取原文。

## 八、如何在当前电脑运行

先进入你下载或克隆后的项目目录：

```text
<你的项目目录>\cstcloud-rag
```

本次已经安装过后端 `.venv` 和前端 `node_modules`。首次真正调用科技云前，仍需自行创建 `.env`，不要把 Token 发给前端或提交到 Git。

### 第一步：配置科技云 Token

打开 PowerShell：

```powershell
cd <你的项目目录>\cstcloud-rag\backend
Copy-Item .env.example .env
notepad .env
```

填写：

```dotenv
CSTCLOUD_API_KEY=你在科技云控制台新生成的Token
CSTCLOUD_BASE_URL=https://uni-api.cstcloud.cn/v1
```

如果任何旧文档里出现过明文 Token，应先到科技云控制台轮换，不要继续使用已经暴露的 Token。

### 第二步：启动后端

当前环境可直接运行：

```powershell
cd <你的项目目录>\cstcloud-rag\backend
.\.venv\Scripts\Activate.ps1
uvicorn app.main:app --reload --port 8000
```

验证：

- 健康检查：`http://localhost:8000/api/health`
- Swagger：`http://localhost:8000/docs`
- 模型列表：`http://localhost:8000/api/models`

看到 `api_key_configured: true` 才表示 Token 被后端读取。

### 第三步：启动前端

另开一个 PowerShell：

```powershell
cd <你的项目目录>\cstcloud-rag\frontend
npm run dev
```

浏览器打开：`http://localhost:5173`。

### 全新电脑安装方式

后端：

```powershell
cd backend
```

访问 `http://localhost:8080`。当前开发机没有 Docker CLI，因此本次完成的是 Compose YAML 解析验证，不要在面试中声称已经在该电脑完成 Docker 镜像实跑。

## 十一、常见运行问题

| 现象 | 原因 | 处理 |
|---|---|---|
| 健康检查 200，但聊天提示 503 | 未配置 Key | 创建 `backend/.env`，重启后端 |
| `/models` 显示 fallback | Key 缺失或远程失败 | 检查 Key、网络与科技云授权模型 |
| 上传文档后状态 failed | Embedding 调用失败或文档无文本 | 查看 error_message；确认 Key 和文件内容 |
| 扫描 PDF 无文本 | pypdf 无法 OCR 图片 | 转成可搜索 PDF，或后续接入 DeepSeek-OCR |
| 切换 Embedding 后检索报维度问题 | 旧索引模型不同 | 点击全库重建，不要复用旧 collection |
| 前端打不开后端 | 后端未启动或端口占用 | 检查 8000 端口和 Vite proxy |
| PowerShell 禁止激活脚本 | ExecutionPolicy 限制 | 临时执行 `Set-ExecutionPolicy -Scope Process Bypass` |
| Vite 提示 chunk 较大 | Element Plus 全量依赖 | 不影响演示；生产改按需导入 |

## 十二、高频面试追问与参考回答

### 1. 为什么不用 LangChain？

这个项目选择直接封装协议，是为了把数据流和异常边界做清楚，避免核心链路藏在框架回调里。未来需要 Agent、复杂工具编排或可观测性时，可以在 service 层之上接 LangGraph，而不改底层数据模型。

### 2. 为什么要混合检索？

向量检索擅长语义相似，但对精确编号、日期、产品型号可能不稳定；BM25 擅长词面匹配，但不理解同义表达。两者结合能提高召回覆盖率。

### 3. 为什么用 RRF，不直接加分？

BM25 分数和余弦相似度不在同一尺度，直接加权会受不同知识库和查询分布影响。RRF 只依赖排名，对尺度变化更稳健。

### 4. Rerank 和 Embedding 有什么区别？

Embedding 把问题和文档分别编码，适合对大量候选做高效召回；Rerank 联合读取问题与候选，计算更贵但相关性判断更细。工程上是“Embedding 召回，Rerank 精排”。

### 5. Query Rewrite 为什么必要？

检索需要一个语义独立的问题。多轮追问里的代词和省略会降低召回，因此先结合最近历史改写，但不把完整历史直接作为检索词。

### 6. 如何防止幻觉？

依靠三层约束：先通过检索和重排提高证据质量；Prompt 要求只依据编号资料、无依据时明确拒答；输出结构化引用，允许用户回看原文。它能降低幻觉，不能保证绝对消除。

### 7. 如何保证引用是真的？

引用不是让模型自由生成，而是后端在检索后根据候选构造 Citation 对象并与回答一起返回。模型正文中的 `[1]` 与后端证据列表使用相同排序，但严格的“正文每个引用都被使用”仍可通过后处理校验继续加强。

### 8. Embedding 模型切换怎么处理？

知识库绑定 Embedding 模型；collection 名加入模型名和哈希。检测到配置不一致时提示全库重建，先删除旧 collection，再重新解析和向量化全部文档。

### 9. 为什么 SSE 使用 POST + fetch？

聊天请求包含问题、会话、知识库和配置，使用 GET 不合适；浏览器原生 EventSource 只能 GET，所以前端用 fetch 读取 `ReadableStream` 并解析 SSE event/data。

### 10. 流式断线为什么不一直重试？

首 token 前重试不会重复内容；已经输出 token 后若重新发送同一请求，模型可能从头生成，前端会看到重复文本。因此代码在产生首 token 后关闭自动重试。

### 11. 为什么 SQLite 和 Chroma 都要存数据？

Chroma负责向量索引；SQLite负责业务实体、原文、BM25、会话、配置和审计。两个系统分别承担检索索引和业务真相源的职责。

### 12. 文档切分有什么不足？

当前是标题、段落、句子优先的基础语义切分，没有使用模型计算语义断点，也没有父子 chunk。后续可增加标题路径、表格感知、parent-child retrieval 和上下文压缩。

### 13. 当前系统如何扩容？

元数据迁移 PostgreSQL；文件进入对象存储；解析与 Embedding 放到 Celery/RQ；向量库换成独立服务；API 多实例部署；引入 Redis 缓存、限流、租户过滤和可观测追踪。

### 14. 目前最大的工程短板是什么？

上传索引当前在请求内同步执行，大文档会占用请求；缺少登录/RBAC 与多租户；扫描 PDF 只预留 OCR；评测仅是基础实现。这些应该主动说明，而不是包装成已完成。

### 15. 怎么做 RAG 评测？

先建立“问题-标准证据-参考答案”数据集。检索看 Recall@K、MRR、nDCG；生成看引用准确率、忠实度、答案相关性；工程看首 token、P50/P95 总延迟和失败率。不同 Embedding/Rerank 组合必须在同一数据集上对比。

### 16. 如果 Rerank 服务失败怎么办？

捕获科技云异常并退回 RRF 排序；Query Rewrite 失败则使用原问题。降级策略确保部分外部能力失败时仍能给出可用结果。

### 17. 如何保护 API Key？

Key 只存在于后端环境变量；前端只请求本地 `/api`；配置对象隐藏 Key 的 repr；日志不记录请求头和文档正文；`.env` 被 `.gitignore` 排除。

### 18. 为什么没有本地部署模型？

项目目标是企业应用工程能力和快速模型切换，因此使用科技云托管模型，避免 GPU 资源和模型运维分散重点。GPU 学习智能体项目则补充了 LoRA 和 vLLM 的模型侧经验。

### 19. 你个人完成了什么？

可以回答：“我完成了需求拆解、API Client、数据模型、解析切片、混合检索、Rerank、Prompt、SSE、会话、Vue 控制台、测试和部署配置。最有价值的三个决策是 RRF 融合、Embedding collection 隔离，以及引用与流式事件分离。”

### 20. 如果再做一周，优先做什么？

第一优先是建立真实评测集；第二是将索引异步化并增加进度；第三是 RBAC/多租户；第四是 OCR 和表格结构化；最后才是更复杂的 Agent。

## 十六、最后的回答策略

面试官问一个模块时，用“问题 - 决策 - 实现 - 取舍 - 下一步”五段式回答：

1. 问题：为什么需要这个模块？
2. 决策：你选择了什么方案？
3. 实现：代码和数据流怎么走？
4. 取舍：它解决了什么，又牺牲了什么？
5. 下一步：生产化会如何改进？

例如回答混合检索：“向量检索对编号召回不稳，所以加入 BM25；两类分数尺度不同，所以用 RRF 按排名融合；当前权重是 0.65/0.35；代价是每次查询要构建当前知识库的 BM25 语料，生产环境会做持久化索引和增量更新。”这比只背技术名词更像真正做过项目的人。
